# Analysis: CPython 3.11 vs PyPy Exception Table Output (Pre-Phase 7)

Generated by comparing `dis.dis` output from `apptest_exctable_dis.py` on CPython 3.11
and untranslated PyPy (exceptiontable branch).

## try/except T as name

**PyPy: 4 entries. CPython: 3 entries.**

CPython's entries:
```
4 to 6 -> 12 [0]          # body
12 to 30 -> 54 [1] lasti  # exc check + STORE_FAST
44 to 52 -> 54 [1] lasti  # cleanup block (None/DELETE/RERAISE)
```

PyPy's entries:
```
2 to 8 -> 10 [0]          # body
10 to 18 -> 42 [1] lasti  # exc check + STORE_FAST
20 to 30 -> 32 [1] lasti  # handler body (POP_EXCEPT path)   <-- EXTRA
32 to 40 -> 42 [1] lasti  # cleanup block
```

The extra PyPy entry (`20 to 30 -> 32`) covers the happy path through the handler body
(the sequence `LOAD_CONST None / STORE_FAST e / DELETE_FAST e / POP_EXCEPT / RETURN_VALUE`).
CPython does `POP_EXCEPT` first then cleans up the name; PyPy does cleanup first then
`POP_EXCEPT`. This different ordering forces PyPy to add exception table coverage over a
range that CPython does not need to cover.

Semantically correct but generates an extra table entry. Known consequence of
`_visit_try_except` code generation order.

## try/except T (no name)

**PyPy: 4 entries. CPython: 3 entries.**

CPython:
```
4 to 6 -> 12 [0]
12 to 30 -> 40 [1] lasti
38 to 38 -> 40 [1] lasti
```

PyPy:
```
2 to 8 -> 10 [0]
10 to 18 -> 30 [1] lasti
20 to 24 -> 26 [1] lasti   <-- EXTRA
26 to 28 -> 30 [1] lasti
```

Same pattern: the extra entry covers the handler body (`POP_EXCEPT / RETURN_VALUE`).
Without a name to clean up there is no reason to cover `POP_EXCEPT` itself.
CPython does not cover it. PyPy generates the entry because it follows a uniform pattern:
every handler body gets coverage. Harmless but wasteful.

## try/except multi-handler

**PyPy: 5 entries. CPython: 3 entries.**

Same pattern repeated for both handlers. Extra entries cover each handler body.
Same root cause as above.

## try/finally

**PyPy: 4 entries. CPython: 2 entries.** Most significant difference.

CPython:
```
4 to 6 -> 12 [0]
12 to 14 -> 16 [1] lasti
```

PyPy:
```
2 to 16 -> 22 [0]        <-- EXTRA outer wrapper
2 to 6 -> 8 [0]          # expected: body -> finally_exc
8 to 10 -> 12 [1] lasti  # expected: finally_exc -> outer_cleanup
22 to 24 -> 26 [1] lasti  <-- EXTRA duplicate
```

The two extra entries point to handler blocks at offsets 22 and 26, which are duplicates of
the blocks at 8 and 12. These duplicates are created by `duplicate_exits_without_lineno`:
the JUMP_FORWARD-to-PUSH_EXC_INFO/RERAISE pattern gets its target copied into a new block
appended at the end of the block list. Both the original `emit_exception_table_entry` calls
and extra entries for the duplicated blocks end up in the table.

This is the Phase 9 known gap. Semantically correct -- if an exception fires in `2 to 16`
it is caught by the duplicate handler which does the same thing -- but wasteful.

## nested try/except inside except handler

**PyPy: 8 entries. CPython: 7 entries.**

PyPy has `20 to 58 -> 60 [1] lasti` covering a wide range including the inner try body
and inner handlers. CPython has more precise entries. The structures are semantically
equivalent; the innermost matching entry wins at lookup time. Harmless overlap.

## Summary

| Pattern              | PyPy entries | CPython entries | Root cause                             |
|----------------------|-------------|-----------------|----------------------------------------|
| try/except as name   | 4           | 3               | Cleanup-before-POP_EXCEPT ordering     |
| try/except no name   | 4           | 3               | Same                                   |
| try/except multi     | 5           | 3               | Same x2                                |
| try/finally          | 4           | 2               | duplicate_exits_without_lineno (Ph. 9) |
| nested               | 8           | 7               | Wide-range entry (overlap harmless)    |

None of these are correctness bugs. Phase 7 (linear SETUP/POP_BLOCK scan) will naturally
fix the handler-body coverage issue by scoping entries to only what SETUP_*/POP_BLOCK pairs
enclose. The try/finally duplicate-block issue requires Phase 9's flow-graph traversal.
